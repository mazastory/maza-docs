# Tistory Duplicate Publish RCA & Prevention Guidelines

## 📌 문제 개요
예약 발행 또는 즉시 발행 시, 동일한 게시물이 10번 이상 중복으로 발행되는 현상이 발생했습니다. 
이 문제는 프론트엔드, 백엔드, 확장 프로그램(Chrome Extension) 간의 통신 비동기성과 티스토리 플랫폼의 특징(방어 로직)이 복합적으로 얽혀 발생한 **Race Condition 및 예외 처리 누락**이 원인이었습니다.

이 문서는 향후 비슷한 구조적 버그가 재발하지 않도록 근본 원인을 분석하고, 개발 시 지켜야 할 가이드라인을 기록합니다.

---

## 🔍 Root Cause Analysis (근본 원인 분석)

중복 발행의 무한 루프는 다음 3가지 버그가 연쇄적으로 작용하여 발생했습니다:

### 1. 확장 프로그램의 `return` 누락에 의한 이중 보고 (Double Reporting)
- **현상**: 익스텐션에서 티스토리에 성공적으로 글을 발행하고 `reportResult`를 호출해 서버에 "성공(success)"을 전송했습니다. 그러나, 메인 실행 함수인 `runTaskOnTab`에서 결과 객체(response)를 `return`하는 코드가 누락되어 있었습니다.
- **결과**: `runTaskOnTab`이 `undefined`를 반환하게 되고, 상위 함수(`executeTask`)는 `undefined`를 받아 또다시 `reportResult(taskId, undefined)`를 호출하려 했습니다. 이때 `TypeError`가 발생하여, `catch` 블록을 타고 서버로 **"실패(failed)"라는 두 번째 잘못된 결과 보고**가 전송되었습니다.

### 2. 서버 검증(Verification) 지연 중 상태 덮어쓰기 (Race Condition)
- **현상**: 서버는 첫 번째 "성공" 보고를 받고 검증(VerificationService) 절차를 시작했습니다. 하지만 검증이 끝나기 전(RSS HTTP 요청으로 1~2초 소요), 익스텐션의 두 번째 "실패" 보고가 서버에 도달했습니다.
- **결과**: 서버의 실패 처리 로직(`handleFailure`)이 동작하여 DB 상태를 강제로 `FAILED`로 덮어썼습니다.

### 3. 티스토리 RSS 차단(404) 및 URL 반환 누락
- **현상 1**: `handleSuccess`에서 검증을 위해 RSS를 파싱하려 할 때, `axios` 요청에 표준 `User-Agent`가 설정되지 않아 티스토리가 봇으로 인식하고 404를 반환했습니다. 이로 인해 RSS를 통한 URL 우회 검증 로직이 마비되었습니다.
- **현상 2**: 익스텐션(`injector.ts`)이 티스토리 발행 완료 후 `/manage/posts`로 리다이렉트된 상황을 감지할 때, 성공했다는 상태(`ALREADY_PUBLISHED`)만 반환하고 정작 중요한 **실제 발행된 URL(`published_url`)을 누락**했습니다.
- **결과**: 서버는 발행된 URL을 알 수 없거나 검증할 수 없어서 3번의 재시도(Retry)를 반복하게 되었고, 이로 인해 익스텐션은 계속해서 글을 새롭게 작성(중복 발행)하게 되었습니다.

---

## 🛡️ 향후 예방 가이드라인 (Prevention Guidelines)

다시는 이런 복합적인 타이밍 버그가 발생하지 않도록 아래 원칙을 엄격하게 준수해야 합니다.

### 1. Extension ↔ Backend 통신 시 명확한 결과 반환 보장
- 익스텐션의 모든 비동기 실행 흐름(`runTaskOnTab`, `executeTask` 등)은 **반드시 명시적으로 결과 객체를 반환**해야 합니다.
- `try-catch` 안에서 외부 API(예: `reportResult`)를 호출할 때 인자가 `undefined`가 될 수 있는 가능성을 항상 체크하고 Optional Chaining(`?.`) 및 기본값 처리를 견고히 해야 합니다.

### 2. 상태 전환(State Transition)의 불변성 (Race Condition 방지)
- 서버에서 상태 업데이트를 진행할 때, 특정 작업이 이미 성공적인 상태(`PUBLISHED` 또는 진행 중인 `VERIFYING`)라면, **후속 실패 보고가 이를 덮어쓰지 못하도록 막아야(Guard)** 합니다. (현재 `handleFailure`에 다운그레이드 방지 로직이 있으나, `VERIFYING` 도중의 끼어들기 처리도 주의해야 함)

### 3. 스크래핑/크롤링 시 식별 가능한 User-Agent 사용
- 티스토리, 워드프레스 등 외부 플랫폼으로 HTTP 요청(`axios`, `fetch`)을 보낼 때는 **반드시 실제 브라우저와 동일한 `User-Agent` 문자열을 포함**해야 합니다. (헤더 누락 시 404, 403, 429 차단 발생)

### 4. 확장 프로그램 상태 감지 시 Payload 완전성 유지
- 페이지 리다이렉트나 예외 상황을 감지해 `ok: true`를 반환할 때는, 후속 로직이 의존하는 필수 데이터(예: `published_url`)가 누락되지 않았는지 꼼꼼히 확인해야 합니다. 빈 값이나 `undefined`가 전송되면 백엔드 검증 로직이 무조건 실패로 처리하게 됩니다.

### 5. RSS 피드 지연과 부적절한 Fallback 로직의 충돌 (무한 검증 지연)
- **현상**: 티스토리는 글 작성 직후 캐싱으로 인해 RSS 피드에 새 글이 바로 반영되지 않는 지연(Delay)이 존재합니다. 이를 우회하기 위해 "방금 쓴 글을 RSS에서 찾을 수 없으면 무조건 성공으로 처리"하는 로직이 있었으나, 그 직전에 "RSS에서 제목을 못 찾으면 무조건 **가장 최근 RSS 글(이전 글)**로 대체(Fallback)한다"는 코드가 존재했습니다.
- **결과**: RSS에 새 글이 없자 이전 글의 URL을 가져오게 되었고, 이전 글의 내용과 새 글의 내용이 다르기 때문에 검증 로직이 "실패"로 판정해 3번씩 무한 검증 대기(티스토리 노출 지연) 상태에 빠졌습니다.
- **예방**: 캐시 지연을 고려한 예외 처리(Bypass)가 있다면, 이를 방해하는 억지스러운 대체(Fallback) 로직은 배제해야 합니다. 존재하지 않는 데이터를 억지로 때워 넣으면 후속 로직 전체가 오염됩니다.

### 6. 자동 재정렬(Stagger) 로직에 의한 에러 메시지 초기화 (Null Error Lockout)
- **현상**: 사용자가 "시간 간격"을 변경하거나 "새로고침(에러 자동 리셋)"을 누르면 대기열을 자동 재정렬(`recover-queue`)합니다. 이때 실패한 태스크(`failed_verification`)는 상태를 그대로 유지해야 하는데, 실수로 모든 태스크의 `last_error`를 `null`로 초기화해버렸습니다.
- **결과**: 에러 메시지(`last_error`)가 `null`이 되면서, 프론트엔드 UI에서 "재시도" 버튼이 숨겨지고 기본 회색 뱃지로 처리되었습니다. 설상가상으로 재시도 조건 로직(`.not('last_error', 'is', null)`)에 걸러져 재시도조차 불가능한 영구 잠금 상태가 되었습니다.
- **조치**: 상태가 유지되는 태스크(keepStatus)는 `last_error`를 초기화하지 않도록 백엔드를 수정했으며, 프론트엔드에서도 에러 텍스트가 없더라도 `failed_verification` 상태라면 무조건 재시도 버튼을 노출하고 초기화 대상에 포함시키도록 방어 로직을 추가했습니다. 또한 `재시도` API 호출 시 `next_retry_at` 시간도 강제로 현재로 갱신하여 백그라운드 재시도 서비스가 즉각 반응하도록 고쳤습니다.

### 7. 대기열 재정렬 시 에러 태스크 양보(Yield) 로직 부재 (길막 현상)
- **현상**: 자동 재정렬(`recover-queue`) 시, 모든 태스크를 단순히 '시간순'으로만 정렬했습니다. 그 결과, 과거에 실패하여 시간이 과거에 멈춰있는 에러 태스크(`failed`, `failed_verification`)들이 무조건 대기열의 최상단을 차지하게 되었습니다.
- **결과**: 실패한 태스크가 대기열의 맨 앞(길목)을 막고 계속해서 재시도와 실패를 반복하는 동안, 정상적으로 발행되어야 할 신선한 새 글들은 뒤로 계속 밀려나는 '길막(Block)' 현상이 발생했습니다. (마치 청소부가 쓰레기를 치운다며 길 한가운데 모아두어 교통 체증을 유발하는 것과 같음)
- **조치**: 큐(Queue) 재정렬 전, 메모리 상에서 태스크를 정렬할 때 **에러가 발생한 태스크를 대기열의 가장 맨 뒤(후순위)로 강제 배치하는 양보(Yield) 방어 로직**을 추가했습니다. 이를 통해 문제 항목이 정상적인 포스팅 발행 흐름을 절대 가로막지 않도록 구조적으로 차단했습니다.
