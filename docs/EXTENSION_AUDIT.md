# MAZA Chrome Extension 코드 점검 리포트 (2026-05-15)

> 이 문서는 사용자로부터 전달받은 아키텍처 점검 리포트로, Maza Studio 익스텐션의 근본적인 구조 개선을 위한 가이드라인으로 활용한다.

## 1. 전체 평가
- **상태**: 기능 확장에 따른 아키텍처 복잡도 임계치 도달.
- **핵심 진단**: "기능 추가"보다 "아키텍처 정리"가 시급한 상태.
- **주요 병목**: 불완전한 책임 분리, 중복 구조, 과도한 MV3 생명주기 대응, 디버깅 난이도 폭증.

## 2. 핵심 문제 TOP 10
1. **높은 결합도**: Extension + Server + Automation Engine이 너무 강하게 엮임.
2. **복잡한 MV3 생명주기**: SW를 억지로 유지하려는 구조 (Offscreen 의존도 높음).
3. **WebSocket 중심 구조**: 브라우저 환경에서 WS 영구 유지는 불안정함.
4. **이중 구조 관리**: `extension/`과 `dist-ext/` 공존으로 인한 빌드 혼선.
5. **Content Script 과의존**: DOM Selector 위주의 로직은 플랫폼 변경에 매우 취약함.
6. **추적 불가능한 런타임 흐름**: 메시지 체인이 너무 길어 에러 추적이 어려움.
7. **약한 타입 경계**: `any` 사용으로 인한 런타임 오류 및 스키마 불일치.
8. **설계 철학 충돌**: "Queue 없음" 원칙과 실제 구현(QueueManager)의 충돌.
9. **서버-익스텐션 경계 모호**: 하나의 레포에 섞여 있어 배포 및 버전 관리 꼬임.
10. **기술 부채 문서의 과잉**: 구조적 복잡도를 문서로 억지 유지 중인 신호.

## 3. 핵심 위험 요소
- **1순위**: MV3 Lifecycle + WebSocket 영속성 (간헐적 실패의 원인).
- **2순위**: 플랫폼 DOM 자동화 의존 (UI 변경 시 대규모 붕괴 위험).
- **3순위**: 상태 관리 복잡도 분산 (Background/Offscreen/Content/Popup/WS).

## 4. 리팩토링 로드맵

### Phase 1 (필수)
- [ ] **메시지 흐름 단순화**: `Server -> Background -> (Target)` 수준으로 축소.
- [ ] **플랫폼 Adapter 분리**: Selector Registry 도입 및 DOM Abstraction.
- [ ] **타입 시스템 강화**: Zod 기반 메시지 스키마 강제.
- [ ] **로그 시스템 통합**: Trace ID 도입 및 중앙 이벤트 로거 구축.

### Phase 2 (고도화)
- [ ] **Extension 최소화**: 익스텐션은 단순 Bridge 역할만 수행.
- [ ] **서버 중심 구조 전환**: 자동화 Orchestration 로직을 서버로 이관.
- [ ] **Workflow Engine 재설계**: 단순 상태 머신 기반으로 단순화.
